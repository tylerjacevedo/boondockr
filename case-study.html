Boondockr

A real-time campsite cancellation notifier built on AWS — designed to catch available sites the moment they open up, before anyone else can.

Python AWS EC2 Lambda API Gateway S3 CloudFront SSM IAM Gmail SMTP AWS CLI
Background

The problem with recreation.gov

Getting a campsite at Yosemite is genuinely difficult. Reservations sell out within seconds of opening each month, and the only tool recreation.gov offers is a basic alert system — capped at 3 alerts, slow to notify, and with no direct link to book when something opens up.

Cancellations happen constantly. The problem isn't availability — it's speed. By the time most people see a notification, the site is already gone. I'd missed sites more than once because of it, so I decided to build something that actually worked.

Architecture

Built entirely on AWS

Every piece of Boondockr runs on AWS — from the Python script polling recreation.gov to the HTTPS-secured control panel. The stack was chosen based on what each layer actually needed, not just familiarity.

Service Role What it does in Boondockr
EC2 Compute Ubuntu t2.micro running 24/7, hosting the Python polling script inside a persistent screen session
Lambda Serverless function Python 3.11 function that receives dashboard commands and executes them on EC2 via SSM — costs nothing when idle
API Gateway REST API Gives Lambda a public URL. Routes /status, /check, /clear, /save-config, and /test-email
S3 Static hosting Hosts the dashboard HTML, configured as a public static website
CloudFront CDN + HTTPS Sits in front of S3, adds SSL, distributes files globally, and handles cache invalidation
SSM Remote execution Lets Lambda securely run shell commands on EC2 without exposing SSH to the internet
IAM Permissions Controls what each service can access. Lambda gets SSM access. EC2 gets an instance profile. Least privilege throughout.
CloudWatch Logging Automatically captures Lambda logs — essential for diagnosing errors in production
Deployment

Everything deployed from the command line

No clicking through the console. Every resource — S3 buckets, CloudFront distributions, Lambda functions, IAM roles, API Gateway, and SSM configuration — was provisioned using the AWS CLI. This kept the setup reproducible and made iterating fast.

# Create and configure S3 website
aws s3 mb s3://boondockr-ui --region us-east-1
aws s3 website s3://boondockr-ui --index-document index.html

# Deploy CloudFront distribution with HTTPS
aws cloudfront create-distribution --origin-domain-name boondockr-ui.s3-website...

# Create IAM roles and attach policies
aws iam create-role --role-name boondockr-lambda-role
aws iam attach-role-policy --policy-arn arn:aws:iam::aws:policy/AmazonSSMFullAccess

# Deploy Lambda and connect to API Gateway
aws lambda create-function --function-name boondockr-api --runtime python3.11
aws apigateway create-rest-api --name boondockr-api
aws apigateway create-deployment --stage-name prod

Using the CLI meant understanding what each resource actually does — there's no wizard to hide the details. Fixing an issue was a single command, not a series of console clicks.

Engineering Challenges

What broke and how I fixed it

Every problem below came up in production. Each one was diagnosed from logs or error output and solved without a guide. That process is where most of the real learning happened.

01

400 errors on every API request

Recreation.gov returned Bad Request on every call with no useful error message.

Browser header fingerprinting

The API silently requires browser-like headers. Added User-Agent, Accept, Referer, and Origin to match what a real browser sends.

02

429 rate limiting — IP blocked

After heavy testing the server IP got flagged and all requests were blocked.

Exponential backoff with jitter

Implemented Tenacity with exponential backoff, Retry-After header support, and randomized delays between requests to look more human.

03

Email hanging indefinitely

Email alerts froze the entire app. Ports 465 and 587 were blocked at the network level by DigitalOcean.

Migrated to AWS EC2

AWS EC2 doesn't block SMTP ports by default. Migrated providers and added a 10-second connection timeout as a safety net.

04

SSM credential errors

The SSM agent was running on EC2 but couldn't authenticate — Lambda had no path to reach it.

IAM instance profile

Created an IAM role with AmazonSSMManagedInstanceCore, built an instance profile, and attached it to the EC2 instance via CLI.

05

CORS blocking dashboard API calls

The frontend showed "Failed to fetch" — the browser was blocking cross-origin requests to API Gateway.

OPTIONS method + CORS headers

Added an OPTIONS method with MOCK integration to API Gateway and configured Access-Control-Allow-Origin on all Lambda responses.

What It Does

Features

Real-time polling

Polls recreation.gov every 2–3 minutes with randomized delays to stay under the radar and avoid rate limiting.

Smart email alerts

One summary email per poll cycle. Available dates are grouped into ranges like May 7–9 rather than a separate notification for each day.

Alert once per site

Each site is tracked persistently on disk. Once alerted, it won't fire again — no repeat notifications across restarts or reboots.

Monthly release protection

Detects when 10+ sites open simultaneously — the monthly release — and sends a single summary instead of flooding the inbox.

Manual re-check

Run --check from the CLI or click a button in the dashboard to force a fresh check on all sites including previously alerted ones.

Live control panel

HTTPS-secured dashboard on S3 and CloudFront. Add campgrounds, manage users, and trigger server commands without SSHing in.

Development History

How it evolved

Each version solved a real problem that only showed up once the previous version was running in production.

v1.0
Core polling engine
Python script with Twilio SMS and SendGrid email. Multi-user config, mass release flood protection, daily digest, and monthly reminders.
v1.1
Cloud migration
Moved from DigitalOcean to AWS EC2 after hitting SMTP port blocking. Replaced Twilio and SendGrid with Gmail SMTP. Added exponential backoff using Tenacity.
v1.2
Smarter notifications
One summary email per poll cycle. Consecutive dates grouped into ranges. 24-hour cooldown per site. Redesigned HTML email template.
v1.3
Persistent alert tracking
Alert history stored to disk. Survives restarts. Manual --check flag added for forcing a re-check on all sites including previously alerted ones.
v2.0
Full-stack AWS dashboard
Static frontend on S3 and CloudFront with HTTPS. Lambda and API Gateway backend. SSM for secure EC2 command execution. Live status, user management, and one-click config updates.
Context

Why I built this

Getting into Yosemite without a reservation is nearly impossible in the summer. The official alert system is slow, limited, and doesn't give you a direct link to book when something opens up. I'd missed sites more than once because of it.

I decided to solve the problem properly. That meant understanding how recreation.gov's API actually works, figuring out how to run something reliably in the cloud, and building a system that would notify me the moment a site opened — not ten minutes later.

The technical stack isn't coincidental. AWS was the right tool for each layer — EC2 for always-on compute, Lambda for lightweight API calls, S3 and CloudFront for a fast and secure frontend. The AWS CLI kept the whole setup reproducible and fast to iterate on.

The best projects come from genuine need. Every technical decision in Boondockr was made in service of a simple goal: get a campsite at Yosemite faster than anyone else. That clarity made the engineering decisions easier and the end result more useful.
